# CharacterStudio 角色编辑器 UI 统一方案（基于当前源码）

> 本方案**以当前 UI 源码为唯一基准**整理，不沿用旧方案中的预设结论。  
> 目标不是重做编辑器，而是在保持现有三栏结构与交互习惯的前提下，**统一视觉语言、收敛高频控件、降低信息拥挤、明确后续渐进改造顺序**。

---

## 1. 方案目标

本方案解决四类问题：

1. 编辑器内部已有多套按钮与面板语言并存，统一度不足
2. 不同页签虽然功能完整，但壳层、标题、工具区、内容区组织不一致
3. 右侧属性区信息密度过高，部分区域仍偏“字段堆叠”
4. 预览页、属性页、属性扩展区仍存在旧式控件风格、硬编码文本与布局裁切风险

本方案**不解决**以下内容：

- 不推翻当前三栏编辑器结构
- 不一次性重做所有业务交互
- 不在第一阶段引入大量新页签或全新导航模型
- 不把角色编辑器改造成底层数据调试器

---

## 2. 当前 UI 架构现状

根据当前源码，角色编辑器已经形成稳定主框架：

- 顶部菜单：`Dialog_SkinEditor.Layout.cs` → `DrawTopMenu()`
- 左栏页签切换：`Dialog_SkinEditor.Layout.cs` → `DrawTabButtons()`
- 中央预览：`Dialog_SkinEditor.Preview.cs` → `DrawPreviewPanel()`
- 右侧属性区：`Dialog_SkinEditor.Properties*.cs`
- 底部状态栏：`Dialog_SkinEditor.Layout.cs` → `DrawStatusBar()`

主窗口组织方式已经明确：

- **左侧**：对象列表 / 图层树 / 子系统面板
- **中间**：角色预览与播放/覆盖控制
- **右侧**：当前选中对象的属性编辑
- **上方**：全局操作
- **下方**：即时状态反馈

因此，当前 UI 的核心问题**不是结构错误**，而是：

1. 统一规范已经出现，但还没有覆盖全部页签
2. 新风格组件和旧风格控件仍混用
3. 高级能力的承载方式需要更清晰的层级化表达

结论：当前 UI 属于 **过渡态（Transitional）**，适合做**渐进统一**，不适合做**推倒重来**。

## 3. 当前可确认的统一基准

以下源码已经构成新的 UI 基准，应作为后续统一参照。

### 3.1 面板壳层基准

代表实现：

- `Dialog_SkinEditor.LayerTree.Panel.cs` → `DrawLayerPanel()`
- `Dialog_SkinEditor.Face.cs` → `DrawFacePanel()`
- `Dialog_SkinEditor.Equipment.cs` → `DrawEquipmentPanel()`
- `Dialog_SkinEditor.BaseAppearance.cs` → `DrawBaseAppearancePanel()`
- `Dialog_SkinEditor.Properties.Panel.cs` → `DrawPropertiesHeader()`

这些实现共同体现出统一的外层结构：

1. 外层深色背景：`UIHelper.PanelFillColor`
2. 1px 边框：`UIHelper.BorderColor`
3. 顶部标题条：`UIHelper.PanelFillSoftColor`
4. 标题底部强调线：`UIHelper.AccentSoftColor`
5. 标题文字：`GameFont.Tiny` + `UIHelper.HeaderColor`
6. 标题下方进入工具区或内容区
7. 内容容器再包一层柔和背景与边框

### 3.2 高频工具按钮基准

代表实现：

- `Dialog_SkinEditor.Layout.cs` → `DrawTopMenu()` 内部按钮
- `Dialog_SkinEditor.LayerTree.Panel.cs` → `DrawIconButton()`
- `Dialog_SkinEditor.Face.cs` → `DrawFaceToolbarButton()`
- `Dialog_SkinEditor.Properties.Panel.cs` → `DrawPropertiesHeaderButton()`

这些按钮的共同特征：

1. 按钮背景使用 `PanelFillSoftColor` 或 `ActiveTabColor`
2. 底部 2px accent 线用于强调状态
3. 悬停时使用 `HoverOutlineColor`
4. 使用 `GameFont.Tiny`、居中文字、弱化文本色
5. 点击逻辑通过 `Widgets.ButtonInvisible(...)` 触发

这套模式应视为**编辑器高频按钮唯一标准**。

### 3.3 列表/滚动区基准

代表实现：

- `DrawLayerPanel()` 的树与自定义图层区
- `DrawEquipmentPanel()` 的装备列表区
- `DrawBaseAppearancePanel()` 的槽位列表
- `DrawFacePanel()` 的内容滚动容器

共同特征：

1. 外层卡片容器包裹滚动区
2. 滚动区使用 `ContractedBy(2f)` 形成内边距
3. 列表项有 hover、selected、alternating row 背景
4. 标题、摘要、工具条与滚动区物理分离

### 3.4 当前统一结论

后续所有 UI 统一工作都必须以这套体系为准：

> **深色外层壳 + 细边框 + 软色标题条 + 底部强调线 + Tiny 字重标题 + 小型工具按钮 + 独立内容容器 + 滚动区内缩**

不得再回退到旧式 `DrawMenuSection + ButtonText` 作为默认视觉语言。

## 4. 统一原则

### 4.1 结构稳定优先

保留现有三栏编辑器框架，不做整体布局翻新。

### 4.2 先统一壳层，再调整信息层级

第一阶段优先统一：

- 面板壳层
- 标题栏
- 工具按钮
- 内容容器

在视觉语言稳定后，再处理字段收纳、摘要层、Advanced 折叠等信息架构问题。

### 4.3 高频交互必须共享同一按钮语义

以下按钮应统一到同一视觉体系：

- 顶部菜单按钮
- 预览控制按钮
- 属性区折叠/展开按钮
- 列表页工具按钮
- 局部上下文按钮

同一编辑器内部不能长期并存“三套以上按钮样式”。

### 4.4 用户可见文本必须统一进入翻译系统

原则：

- 所有用户可见标签、按钮、提示、空态文本必须走 key
- 不直接暴露内部字段名给最终用户
- 技术值可保留原值，但标签必须可翻译

### 4.5 高级能力不能继续线性堆叠

复杂能力后续应按层级表达：

- 主字段：直接编辑
- 摘要信息：简短展示当前状态
- 高级字段：折叠区
- 复杂系统：摘要 + 编辑按钮 + 子窗口

### 4.6 不为了统一而破坏既有工作流

统一方案必须尽量做到：

- 不改变现有功能入口位置
- 不重排已形成肌肉记忆的核心操作路径
- 不因视觉收敛而降低编辑效率

---

## 5. 各区域统一规范

## 5.1 顶部菜单

当前状态：

- `DrawTopMenu()` 已是成熟基准

后续要求：

1. 顶部菜单继续作为全局按钮视觉标准
2. 新增全局按钮必须沿用现有 fill / border / accent / hover 逻辑
3. 不允许在顶部菜单引入系统默认 `ButtonText(...)`

结论：**顶部菜单无需重做，仅作为标准输出源。**

---

## 5.2 左侧页签区

当前状态：

- 主体布局稳定
- `BaseAppearance / Layers / Face / Weapon / Equipment / Attributes` 已固定

后续要求：

1. 左栏各页必须统一遵循“标题栏 + 工具区/摘要区 + 内容区”三段式
2. 列表型页签优先采用工具条 + 滚动列表模式
3. 内容型页签优先采用摘要/Section + 滚动内容模式

---

## 5.3 BaseAppearance 面板

当前状态：

- 已基本接入新壳层风格
- 但摘要文本仍有直写中文：`已启用槽位...`
- 行内启用/编辑交互仍带有较强符号化倾向（`◆ / ◇ / ◉ / ◯ / ···`）

统一要求：

1. 保留当前外层卡片结构
2. 摘要文案改为翻译 key
3. 行操作图标保留可行，但 tooltip 与状态语义要补完整
4. 选中态、hover 态、禁用态继续沿用当前统一色板

结论：**BaseAppearance 不属于重构对象，属于细化与收口对象。**

---

## 5.4 LayerTree 面板

当前状态：

- 是当前最成熟的统一样板之一
- 具备完整标题条、工具条、滚动区、列表行交互体系

统一要求：

1. 作为左栏列表型面板首要参考实现
2. 后续若抽公共工具按钮方法，应优先从这里提炼
3. 自定义图层区与树区的分隔样式可继续作为 section 分隔基准

结论：**LayerTree 作为列表型页面样板，不优先改。**

---

## 5.5 Face 面板

当前状态：

- 已形成相对成熟的内容型页面规范
- 具备工具按钮、信息 banner、section title、滚动内容区
- 但局部仍可能存在零散直写字符串，需要继续 key 化

统一要求：

1. Face 页作为“复杂内容型页面”的主要样板
2. `DrawFaceToolbarButton()` 与 `DrawFaceInfoBanner()` 的设计应沉淀为可复用模式
3. 后续复杂子系统页签优先参考 Face 页的 section 组织方式，而非旧式属性堆叠

结论：**Face 是内容型统一基准之一。**

---

## 5.6 Equipment 面板

当前状态：

- 标题、工具条、列表容器统一度较高
- 列表项已具备 hover / selected / 危险按钮等较完整状态

统一要求：

1. Equipment 作为“工具条 + 实体列表”页签基准
2. 危险按钮、上下文菜单、行摘要模式可继续推广到同类列表页
3. 若后续统一图标按钮尺寸，应优先保持与此页兼容

结论：**Equipment 是左栏实体列表页的良好参考。**

---

## 5.7 Attributes 面板

当前状态：

- 已接入外层新壳层
- 但内部仍偏“字段线性排列 + 局部原生按钮”
- LLM 区域、自定义属性 buff 区与基础属性区的视觉层次仍不够统一

存在问题：

1. 内容组织偏平铺
2. LLM 输入区仍更像功能块拼接，而不是统一 section
3. `Add / Clear / Search` 一类按钮仍应继续向统一工具按钮语言收敛

统一要求：

1. Attributes 页保留当前功能结构，但补足 section 层级
2. 将基础属性、数值属性、Buff 编辑、LLM 辅助拆成明确块级区域
3. 工具按钮逐步替换为统一按钮绘制方式
4. 文本区域、搜索区域、按钮条外框统一为内容卡片或 section box

结论：**Attributes 是第一批需要继续收敛的页面。**

---

## 5.8 Preview 面板

当前状态：

- 功能密度高，是中央工作区核心
- 但也是当前最需要统一的区域之一
- 旧式按钮、覆盖控制、播放控制、状态预设混在同一区域时，视觉语义不够稳定

统一要求：

1. 预览面板必须整体切换到当前统一卡片壳层
2. 所有高频控制按钮统一到 toolbar button 体系
3. 控制区按用途拆分为：
   - 画布控制
   - 播放/流程控制
   - 覆盖控制
   - 状态摘要/提示
4. 预览画布与控制区之间必须形成明确分层
5. 所有预设、状态、通道文案必须走翻译 key

结论：**Preview 是 UI 统一工作的最高优先级区域。**

---

## 5.9 右侧 Properties 区

当前状态：

- Header 已接入较新的标题与按钮风格
- 但整体仍承担过多字段密度与复杂度
- 折叠 section 已存在，但组织语义还不够清晰

统一要求：

1. 保留当前右栏职责：编辑当前选中对象属性
2. 不再把右栏继续扩张成无限增长的大表单
3. 后续右栏按四层结构整理：
   - **Basic**：高频核心字段
   - **Context**：对象状态/摘要/关联信息
   - **Advanced**：中频高级字段
   - **Tools**：只放按钮与摘要入口
4. 复杂系统不在主右栏全文展开，优先改为“摘要 + 编辑入口”
5. 顶部 `+ / -` 展开折叠按钮保留，但统一 tooltip、语义与可达性

结论：**Properties 是信息架构收敛的主战场。**

---

## 6. 文本与本地化规范

### 6.1 必须 key 化的内容

以下内容不得继续直写：

- 标题
- 按钮文本
- tooltip
- 空状态说明
- 摘要行文案
- 状态切换文字
- 技术窗口中的中文标签

### 6.2 可保留符号，但不能只靠符号传达意义

如 `+ / - / ↑ / ↓ / ↻ / ◉ / ◯ / ◆` 可以保留，但必须满足：

1. 有 tooltip
2. 状态足够清晰
3. 不成为唯一语义载体

### 6.3 技术值展示原则

如果必须显示 defName、内部字段名、路径等技术值，应采用：

- 中文标签 + 技术值

而不是直接把技术字段名当用户文案使用。

---

## 7. 布局与密度规范

### 7.1 不再默认假设单行可承载所有属性

以下类型内容允许拆成双行或分块：

- 长中文标签
- 标签 + 输入框 + 按钮组合
- 多个相关数值输入
- 带摘要说明的复杂属性

### 7.2 摘要优先于展开

复杂对象优先展示摘要，例如：

- 已配置方向覆盖
- 已绑定能力 N 项
- 已启用动画规则 N 条
- 已设置颜色/遮罩/高级材质

仅在用户需要时进入详细编辑。

### 7.3 滚动区必须保留视觉缓冲

统一要求：

- 滚动内容外必须有容器边界
- 内容不直接贴边滚动
- section 之间留固定间距

---

## 8. 统一改造优先级

## P0：立即统一

1. **Preview 面板整体切换到统一壳层与按钮语言**
2. **所有用户可见硬编码文案与乱码文本清理为翻译 key**
3. **Properties 高层标题按钮语义统一并补全 key 化文本**

## P1：结构性收敛

1. **Attributes 面板补齐 section/card 层级**
2. **BaseAppearance / 其他细分列表页清理直写摘要文本**
3. **右栏 Properties 按 Basic / Context / Advanced / Tools 做结构整理**

## P2：中期优化

1. 将复杂系统改为“摘要 + 按钮 + 子窗口”
2. 抽取公共 Panel Shell / Toolbar Button / Section Box 绘制帮助方法
3. 进一步统一技术窗口（导出器、文件浏览器、检查器、性能统计等）

---

## 9. 推荐落地顺序

### 阶段一：视觉统一

目标：用户立即感受到编辑器“像一个整体”。

顺序：

1. Preview
2. Properties Header 与右栏顶层壳层
3. Attributes
4. 次级窗口的文本 key 化

### 阶段二：信息收敛

目标：降低右栏和复杂页面的拥挤感。

顺序：

1. Properties 四层结构
2. Attributes 分区卡片化
3. 复杂能力摘要层

### 阶段三：公共抽象

目标：减少重复 UI 代码，防止重新分叉。

顺序：

1. 抽公共 panel shell
2. 抽公共 toolbar button
3. 抽公共 summary/banner/section box
4. 建立统一文案与 key 命名规则

---

## 10. 实施约束

后续所有 UI 统一实现必须遵守：

1. **不改三栏总布局**
2. **不一次性重写全部页签**
3. **优先小步渐进替换**
4. **视觉统一先于大规模交互重构**
5. **高频按钮与标题栏必须先统一**
6. **所有用户可见文本必须可本地化**
7. **复杂能力优先做摘要入口，不直接铺满主面板**

---

## 11. 最终统一标准

当后续 UI 工作达到以下状态时，可视为完成统一：

- 同一编辑器内部不再出现明显割裂的旧式按钮风格
- 各页签都遵循“标题栏 + 工具/摘要区 + 内容区”结构
- 右侧属性区不再是无层级大表单
- 预览区成为统一视觉语言的一部分，而不是独立旧模块
- 所有可见文本进入翻译系统
- 新增页面默认复用公共 UI 规则，而不是重新发明样式

---

## 12. 一句话结论

CharacterStudio 当前 UI **不需要推倒重做**。  
正确路线是：

> **以现有 Layer / Face / Equipment / TopMenu / PropertiesHeader 这套新风格为统一基准，优先收敛 Preview、Attributes、Properties 的视觉语言与信息层级，再逐步抽象公共组件，完成全编辑器统一。**
