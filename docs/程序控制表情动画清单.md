# 程序控制表情动画清单

## 说明

本文档按当前 `CharacterStudio` 已接入的运行时代码整理“程序控制表情动画”能力，重点回答下面几个问题：

- 当前**哪些部件**会被程序驱动；
- 它们分别由什么状态量驱动；
- 当前是通过**切图**、**UV 偏移**，还是**额外程序形变**来表现；
- 哪些部件已经具备 **Left / Right 左右独立资源与节点基础**；
- 编辑器预览里哪些项目可以手动强制覆盖。

> 注意：
>
> - 本文档只写“当前代码里已经实现并接入”的能力，不写未来设计。
> - 是否最终能看到效果，仍取决于当前皮肤是否提供了对应资源。
> - 这里的“程序控制”既包括运行时自动状态，也包括编辑器预览里的强制覆盖。
> - 本文档重点是“部件级能力”，不再只列 `ExpressionType` 枚举。

---

## 一、当前控制链总览

当前脸部系统主要由 3 层控制量组成：

1. **ExpressionType**：整套表情状态，如 `Happy / Sad / AttackRanged / Dead / Blink`
2. **通道状态**：由表情进一步解析出的  
   `MouthState / LidState / BrowState / EmotionOverlayState`
3. **EyeDirection**：眼睛注视方向  
   `Center / Left / Right / Up / Down`

它们在代码中的关系如下：

| 控制量 | 运行时来源 | 预览是否可强制 | 主要作用对象 | 当前实现方式 |
|---|---|---:|---|---|
| `ExpressionType` | [`CompPawnSkin.UpdateExpressionState()`](../Source/CharacterStudio/Core/CompPawnSkin.cs) + `Blink` 内部逻辑 | 是 | 整脸贴图、Layered Base、各分层部件的 expression 资源选择 | 整脸切图 / 分层部件路径切换 |
| `EyeDirection` | [`CompPawnSkin.UpdateEyeDirectionState()`](../Source/CharacterStudio/Core/CompPawnSkin.cs) | 是 | 旧眼方向覆盖层、layered pupil、任何配置为 Eye 通道的图层 | 方向贴图切换或 UV 偏移 |
| `MouthState` | [`ResolveMouthState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs) | 是 | `Mouth` | 状态后缀切图 + 程序形变 |
| `LidState` | [`ResolveLidState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs) | 是 | `Eye / UpperLid / LowerLid` | 状态后缀切图 + 程序形变 |
| `BrowState` | [`ResolveBrowState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs) | 是 | `Brow` | 状态后缀切图 + 程序形变 |
| `EmotionOverlayState` | [`ResolveEmotionOverlayState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs) | 是 | `Blush / Tear / Sweat / Overlay` | 显隐 + 状态切图 + 程序形变 |
| `Blink` | [`UpdateBlinkLogic()`](../Source/CharacterStudio/Core/CompPawnSkin.cs) | 可直接预览指定 | 眼睑、眼部、整脸闭眼态 | 短时表情状态覆盖 |

## 二、部件级能力总表

下表是当前最重要的“哪个部件可以怎么动”总览：

| 部件 | 主要驱动量 | 当前具体表现 | 左右独立资源/节点 | 预览可手动强制 |
|---|---|---|---|---|
| 整脸 Head / FullFace | `ExpressionType` | 整张头部贴图切换；支持 `_north / _east` 朝向变体；支持整脸多帧表情 | 否 | 表情 |
| Layered Base | `ExpressionType` | 分层脸的底图切换；在 World Track 下作为保底脸显示 | 否 | 表情 |
| `Brow` | `BrowState` | 眉毛状态切图 + 程序旋转/位移/缩放 | 是 | 眉毛状态 |
| `Eye` | `LidState` | 眼白/眼型随闭眼、半闭眼、笑眼等状态变化；也可吃到特殊闭眼 expression 资源 | 是 | 眼睑状态 |
| `UpperLid` | `LidState` | 上眼睑开合、半闭、笑眼等；带程序位移/缩放 | 是 | 眼睑状态 |
| `LowerLid` | `LidState` | 下眼睑跟随 Blink / Close / Half / Happy 等状态 | 是 | 眼睑状态 |
| `Pupil` | `EyeDirection` | 推荐为 UV 偏移移动瞳孔；非 UV 模式下也可做方向贴图切换 | 是 | 眼方向 |
| `Mouth` | `MouthState` | 嘴型切图 + 程序形变（微笑、张口、下垂、睡眠口型等） | 否 | 嘴型状态 |
| `Blush` | `EmotionOverlayState` / 正向表情 | 显隐 + 轻微脉动 | 通常否 | 情绪覆盖状态 |
| `Tear` | `EmotionOverlayState` / 负向表情 | 显隐 + 轻微下坠 / 波动 | 通常否 | 情绪覆盖状态 |
| `Sweat` | `EmotionOverlayState` / 战斗惊慌疼痛 | 显隐 + 摆动 | 通常否 | 情绪覆盖状态 |
| `Overlay` | `EmotionOverlayState` | 泛用情绪覆盖层；根据 overlayId 映射为具体情绪部件或通用层 | 通常否 | 情绪覆盖状态 |

## 三、各部件当前到底“怎么动”

## 1. 整脸 Head / FullFace

对应代码主入口：[`PawnRenderNodeWorker_FaceComponent`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_FaceComponent.cs)

### 当前已实现能力

- 按 `ExpressionType` 切整张头部贴图；
- 支持 `_north` / `_east` 朝向变体；
- 支持整脸表情多帧动画；
- 当 `workflowMode != LayeredDynamic` 时，肖像轨会直接由这个 worker 驱动整脸；
- 当当前轨道是 `World Track` 时，会优先取 world track 编译路径。

### 需要注意

- 在 `LayeredDynamic` 肖像轨中，`FaceComponent` **不会**再重复绘制肖像底脸，避免和 `[Face] Base` 双绘；
- 所以分层脸模式下，**整脸切图不是主链**，主链是自定义 `[Face]` 图层节点；
- 整脸模式不支持左右独立眉眼嘴。

---

## 2. Layered Base

对应代码主入口：[`PawnRenderNodeWorker_CustomLayer`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs)

### 当前已实现能力

- `Base` 作为分层脸底图，可以按 `ExpressionType` 切换；
- 在 `Portrait Track` 中，`Base` 会作为 `[Face] Base` 自定义图层参与渲染；
- 在 `World Track` 中，局部脸部节点会被关闭，仅保留 `Base` 作为保底世界轨脸。

### 需要注意

- 当前 World Track 的目标是“保底可用”，不是完整分层动态脸；
- 所以真正丰富的眉眼嘴动画，主要仍体现在肖像轨。

---

## 3. Brow（眉毛）

对应驱动：

- 状态来源：[`ResolveBrowState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)
- 渲染入口：[`ApplyProgrammaticBrowTransform(...)`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs)

### 当前已实现能力

`Brow` 会同时吃到两类控制：

1. **资源层切换**
   - 可按 `BrowState` 找 `Normal / Angry / Sad / Happy` 等状态后缀资源；
2. **程序形变**
   - `Angry`：眉毛整体更压低、更收紧；
   - `Sad`：眉毛更上挑/下垂；
   - `Happy`：轻微压缩和上扬；
   - `Normal`：保留极轻微呼吸式微动。

### 左右独立情况

- `Brow` 已属于支持 side-specific 的 paired 部件；
- 当前主链已经支持 `Left / Right` 独立资源与独立渲染节点；
- 这意味着左右眉毛**资源层面**已经可以不同。

### 当前限制

- 运行时状态仍然只有一个全局 `BrowState`；
- 也就是说：**左右资源与节点独立已具备，但 LeftBrowState / RightBrowState 这种独立状态 API 还没有。**

---

## 4. Eye / UpperLid / LowerLid（眼部与眼睑）

对应驱动：

- 状态来源：[`ResolveLidState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)
- 渲染入口：[`ApplyProgrammaticLidTransform(...)`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs)

### 4.1 Eye（眼白 / 眼型底层）

当前代码里，`Eye` 这个 layered part **不是主要跟 EyeDirection 走**，而是映射到 `LidState` 通道。

也就是说，`Eye` 更像“眼型/眼白基础层”，它当前主要用于：

- `Normal`
- `Blink`
- `Half`
- `Close`
- `Happy`

这些“眼睑/眼型”状态，而不是瞳孔注视方向本身。

### 4.2 UpperLid / LowerLid

`UpperLid` 和 `LowerLid` 是当前最直接的程序化眼部动画层。

#### `UpperLid` 当前表现

- `Blink`：明显下压，接近闭眼；
- `Close`：持续闭眼；
- `Half`：半睁半闭；
- `Happy`：轻微笑眼压低；
- `Normal`：保持基础位置，附带极轻微微动。

#### `LowerLid` 当前表现

- `Blink`：轻微上抬/压缩；
- `Close`：闭眼态辅助；
- `Half`：半闭眼辅助；
- `Happy`：轻微笑眼变化；
- `Normal`：极弱微动。

### 特殊闭眼资源优先

对于 `Eye / Pupil / UpperLid / LowerLid` 这几类部件，当前代码会对以下表情特殊处理：

- `Blink`
- `Sleeping`
- `Dead`

如果这些 expression 自己配置了专门部件资源，会优先取这些“闭眼/闭目态资源”。

### 左右独立情况

这几类都是当前已经支持 side-specific 的 paired 部件：

- `Eye`
- `Pupil`
- `UpperLid`
- `LowerLid`

所以当前已经具备：

- 左右独立资源；
- 左右独立节点；
- 左右独立编辑层。

### 当前限制

这套基础已经足够支撑“wink / 单眼动作”的资源结构，但要分清两个层面：

1. **资源/节点层面**
   - 已经支持左右独立；
   - 同一个 expression 下，左右两边可以引用不同资源；
2. **运行时状态层面**
   - 当前仍然只有一个全局 `LidState`；
   - 还没有 `LeftLidState / RightLidState` 这种独立状态输入。

所以当前主线是：

- **已经有单眼动作的资源基础与渲染基础**；
- 但若要做“程序实时控制左眼闭、右眼开”的细粒度逻辑，后续还需要独立左右状态 API。

---

## 5. Pupil（瞳孔）

对应驱动：

- 状态来源：[`CurEyeDirection`](../Source/CharacterStudio/Core/CompPawnSkin.cs)
- 渲染入口：
  - [`PawnRenderNodeWorker_CustomLayer.GetMaterialPropertyBlock(...)`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs)
  - [`PawnRenderNodeWorker_EyeDirection`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_EyeDirection.cs)

### 当前代码里实际上有两套“眼方向”系统

#### A. 旧眼方向覆盖层（整图/旧配置兼容）

`PawnRenderNodeWorker_EyeDirection` 负责一套“额外眼方向覆盖层”：

- 可按 `texCenter / texLeft / texRight / texUp / texDown` 做方向贴图切换；
- 也支持 UV 偏移模式；
- 但它在以下情况会隐藏：
  - `World Track`
  - `Blink`
  - `Sleeping`
  - `Dead`

这套更偏向整图/旧配置兼容。

#### B. Layered pupil（当前 layered 工作流主线）

对于分层脸，当前更推荐、也更符合现有导入主线的是：

- `Pupil` 保留一张基础贴图；
- 通过 `EyeRenderMode.UvOffset` + `eyeUvMoveRange / pupilMoveRange` 做 UV 偏移；
- 根据 `EyeDirection` 修改 `_MainTex_ST` 的 offset 部分，让瞳孔在贴图内部移动。

### 当前 layered pupil 能怎么动

- `Center`：不偏移；
- `Left`：UV 向右偏移，视觉上瞳孔向左看；
- `Right`：UV 向左偏移；
- `Up`：UV 向下偏移；
- `Down`：UV 向上偏移。

### 当前 layered pupil 还支持什么

如果没有走 UV 模式，代码仍然支持：

- 按方向后缀资源切换；
- 或通过 Eye 通道状态去找方向对应资源。

但就当前 layered 主线来说，**推荐路径已经是 UV-only pupil**。

### 左右独立情况

- `Pupil` 已支持 `Left / Right` 独立资源与独立节点；
- 但当前 `EyeDirection` 仍是一个全局方向值；
- 所以默认情况下左右瞳孔会**一起朝同一个方向移动**。

---

## 6. Mouth（嘴）

对应驱动：

- 状态来源：[`ResolveMouthState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)
- 渲染入口：[`ApplyProgrammaticMouthTransform(...)`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs)

### 当前已实现能力

`Mouth` 当前既支持状态切图，也支持程序形变：

- `Smile`：更宽、更平、更上扬；
- `Open`：明显张口，垂直方向拉伸；
- `Down`：下垂/苦脸；
- `Sleep`：更扁、更收缩；
- `Normal`：基础嘴型。

### 额外特殊行为

- 即使是 `Normal`，如果当前表情是 `Eating`，代码也会额外加入“咀嚼感”振动。

### 当前限制

- 嘴当前不是 paired side-specific 部件；
- 也就是没有左右半嘴独立控制主链。

---

## 7. Blush / Tear / Sweat / Overlay（情绪覆盖层）

对应驱动：

- 状态来源：[`ResolveEmotionOverlayState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)
- 渲染入口：[`ApplyProgrammaticEmotionTransform(...)`](../Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs)

### 当前已实现能力

这类部件当前不是简单“有/无”，而是会：

- 根据情绪状态自动出现或隐藏；
- 使用 alpha 渐入渐出；
- 叠加轻微程序化位移 / 缩放 / 摆动。

### 分项表现

#### `Blush`

在这些情况更容易出现：

- `EmotionOverlayState.Blush`
- `EmotionOverlayState.Lovin`
- `Happy`
- `Cheerful`
- `Lovin`

表现为轻微脉动、轻微放大。

#### `Tear`

在这些情况更容易出现：

- `EmotionOverlayState.Tear`
- `EmotionOverlayState.Gloomy`
- `Gloomy`
- `Sad`
- `Hopeless`
- `Pain`

表现为轻微下坠、微弱脉动。

#### `Sweat`

在这些情况更容易出现：

- `Scared`
- `Pain`
- `AttackMelee`
- `AttackRanged`
- `WaitCombat`

表现为轻微摇摆和抖动感。

#### 通用 `Overlay`

- 这是泛用情绪覆盖层；
- 如果 overlayId 被映射为具体情绪部件，会按对应类型走；
- 否则按“泛用 Overlay”逻辑显隐与轻微形变。

### 当前限制

- 当前这类覆盖层通常不走左右独立主链；
- 它们主要是整块情绪效果层，而不是成对器官层。

## 四、通道状态映射：程序到底如何决定“怎么动”

下面这部分是“表情 → 部件状态”的实际映射规则。

## 1. 嘴型联动：`ExpressionType -> MouthState`

来源：[`ResolveMouthState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)

| 表情 | 解析结果 |
|---|---|
| `Happy / Cheerful / Lovin / SocialRelax` | `Smile` |
| `Eating / AttackMelee / AttackRanged / Scared` | `Open` |
| `Gloomy / Sad / Hopeless / Pain / Tired` | `Down` |
| `Sleeping / LayDown / Dead` | `Sleep` |
| 其他 | `Normal` |

---

## 2. 眼睑联动：`ExpressionType -> LidState`

来源：[`ResolveLidState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)

| 表情 | 解析结果 |
|---|---|
| `Blink` | `Blink` |
| `Sleeping / Dead` | `Close` |
| `Tired / Gloomy / Sad / Hopeless / Pain / LayDown` | `Half` |
| `Happy / Cheerful / Lovin` | `Happy` |
| 其他 | `Normal` |

> 当前 `Eye / UpperLid / LowerLid` 都会吃这条通道，  
> 所以“闭眼、半闭眼、笑眼”主要是这条链驱动出来的。

---

## 3. 眉毛联动：`ExpressionType -> BrowState`

来源：[`ResolveBrowState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)

| 表情 | 解析结果 |
|---|---|
| `Angry / WaitCombat / AttackMelee / AttackRanged` | `Angry` |
| `Gloomy / Sad / Hopeless / Tired / Pain / Dead` | `Sad` |
| `Happy / Cheerful / Lovin` | `Happy` |
| 其他 | `Normal` |

---

## 4. 情绪覆盖联动：`ExpressionType -> EmotionOverlayState`

来源：[`ResolveEmotionOverlayState(...)`](../Source/CharacterStudio/Core/CompPawnSkin.cs)

| 表情 | 解析结果 |
|---|---|
| `Lovin` | `Lovin` |
| `Happy / Cheerful / SocialRelax` | `Blush` |
| `Scared / Shock / WaitCombat / AttackMelee / AttackRanged` | `Sweat` |
| `Pain / Sad / Hopeless / Dead` | `Tear` |
| `Gloomy` | `Gloomy` |
| 其他 | `None` |

> 这是当前代码里已经接入的真实映射。  
> 因此现在的情绪覆盖并不只有 `Lovin / Gloomy` 两类，而是已经扩展到了 `Blush / Sweat / Tear / Gloomy / Lovin`。

---

## 5. 眼方向联动：`Pawn 状态 -> EyeDirection`

来源：[`UpdateEyeDirectionState()`](../Source/CharacterStudio/Core/CompPawnSkin.cs)

### 当前推断优先级

1. `Dead / Downed / InBed` → `Center`
2. 有 Job 目标格 → 根据目标相对方向推断 `Left / Right / Up / Down`
3. 没有目标格时 → 根据当前朝向推断
   - `East` → `Right`
   - `West` → `Left`
   - `North` → `Up`
   - 其他 → `Center`

也就是说：

- 这条链会影响旧眼方向覆盖层；
- 也会影响 layered pupil 的 UV 偏移。

## 五、当前运行时自动接入的表情状态

以下表情已经在 [`CompPawnSkin.UpdateExpressionState()`](../Source/CharacterStudio/Core/CompPawnSkin.cs) 中明确接入自动逻辑。

| 表情 | 触发条件 | 代码来源 |
|---|---|---|
| `Dead` | Pawn 死亡 | `pawn.Dead` |
| `Pain` | Pawn 倒地 | `pawn.Downed` |
| `Sleeping` | 在床上/休息 | `RestUtility.InBed(pawn)` |
| `Eating` | 正在进食 | `JobDefOf.Ingest` |
| `Lovin` | 正在亲密互动 | `JobDefOf.Lovin` |
| `LayDown` | Job 名匹配 `LayDown` | `job.defName` |
| `Strip` | 脱装备相关 Job | `JobDefOf.Strip` / 名称匹配 |
| `Hauling` | 搬运/携带相关 Job | `HaulToCell / HaulToContainer / 名称匹配` |
| `Reading` | 阅读/学习相关 Job | 名称匹配 `Read / Study` |
| `SocialRelax` | 社交放松 | 名称匹配 `SocialRelax / Chat` |
| `Working` | 制作、研究、清洁、建造、种植等工作 | 多个工作类 Job 名称匹配 |
| `WaitCombat` | 战斗警戒 | 名称匹配 `WaitCombat` |
| `AttackMelee` | 近战攻击 | 名称匹配 `AttackMelee` |
| `AttackRanged` | 远程攻击 | 名称匹配 `AttackStatic / Shoot / Burst` |
| `Scared` | 惊慌/逃跑类精神状态 | `MentalStateDef` 名称匹配 `Flee / Panic / WildMan` |
| `Angry` | 攻击性精神状态 | 非恐慌型精神状态分支 |
| `Tired` | 休息值低于 0.15 | `rest < 0.15f` |
| `Hopeless` | 心情低于 0.1 | `mood < 0.1f` |
| `Sad` | 心情低于 0.2 | `mood < 0.2f` |
| `Gloomy` | 心情低于 0.4 | `mood < 0.4f` |
| `Cheerful` | 心情高于 0.9 | `mood > 0.9f` |
| `Happy` | 心情高于 0.8 | `mood > 0.8f` |
| `Neutral` | 以上条件都不满足时回退 | 默认分支 |

### 当前未自动赋值的表情

虽然存在于 `ExpressionType` 枚举里，但目前没有在主运行时自动切换函数里直接赋值：

- `Shock`
- `Goto`

它们当前仍然可以：

- 作为资源项存在；
- 在编辑器预览中被直接指定；
- 被其他未来逻辑接入。

## 六、Blink（眨眼）是独立的程序动画

来源：[`UpdateBlinkLogic()`](../Source/CharacterStudio/Core/CompPawnSkin.cs)

`Blink` 与普通表情不完全一样，它当前有 3 重身份：

1. **运行时自动眨眼**
2. **预览器可手动指定**
3. **自动播放预览序列中的内置状态**

### 当前自动眨眼逻辑

- 只有在**没有表情预览覆盖**时才会自动触发；
- `Sleeping` 与 `Dead` 时不会触发；
- 命中概率后会进入短暂 `Blink` 状态；
- 持续若干 Tick 后恢复，并请求刷新渲染。

### 对渲染的影响

- `LidState` 会被解析成 `Blink`；
- `Eye / UpperLid / LowerLid` 会优先进入闭眼资源/形变；
- 旧眼方向覆盖层会隐藏，避免叠在闭眼贴图上。

## 八、当前最关键的实现边界与注意事项

## 1. Left / Right 独立已经到哪一步了

当前已经完成的是：

- `Brow / Eye / Pupil / UpperLid / LowerLid`
  - 支持左右独立资源；
  - 支持左右独立节点；
  - 支持左右独立编辑层。

但当前**还没有**做到的是：

- `LeftLidState / RightLidState`
- `LeftBrowState / RightBrowState`
- `LeftEyeDirection / RightEyeDirection`

所以要准确理解为：

- **资源与节点层面的左右独立基础已经有了；**
- **运行时输入状态仍然大多是单值全局状态。**

---

## 2. Layered pupil 与旧 eye direction overlay 是两套系统

当前代码里不要把这两套混为一谈：

### 旧系统

- `PawnRenderNodeWorker_EyeDirection`
- 适合整图/旧配置兼容
- 可做方向贴图替换，也可做 UV 偏移

### LayeredDynamic 主线

- `Pupil` 自己就是分层部件
- 推荐只保留基础 pupil 贴图
- 用 UV 偏移来做注视移动

对于 layered 脸，当前主线应理解为：

- **眼睛朝向主要由 pupil 自己控制**
- **不是再额外叠一层旧眼方向整图覆盖**

---

## 3. World Track 与 Portrait Track 的差异非常大

当前运行时是双轨思路：

### `Portrait Track`

- 分层脸局部节点会正常工作；
- 眉、眼、眼睑、嘴、情绪覆盖层都会显示；
- 程序形变、alpha 渐变、UV 偏移等都在这里体现得最完整。

### `World Track`

- 当前是“保底脸”策略；
- 除 `Base` 外，其余局部脸部节点会被关闭；
- 旧眼方向覆盖层也会关闭。

所以如果测试时发现：

- 世界里表情层级不如预览里丰富；
- 眉眼嘴局部动画主要在肖像里更明显；

这是当前实现的预期行为。

---

## 4. `Eye` 当前更像“眼型层”，不是“注视方向层”

这点很容易误解。

在当前 layered face 主链中：

- `Eye` 主要跟 `LidState`；
- `Pupil` 才是主要跟 `EyeDirection` 的部件。

因此如果要做“看左看右”，应优先配置：

- `Pupil`
- `eyeUvMoveRange / pupilMoveRange`

而不是把 gaze 方向逻辑塞进 `Eye`。

---

## 5. 整脸多帧动画已接入；分层程序动画以状态形变为主

当前确认已经接入的是：

- **整脸表情多帧切换**：由 `FaceComponent` 按 `animTick` 取帧；
- **分层部件程序形变**：通过 `CustomLayer` 对眉、眼睑、嘴、情绪层做旋转/位移/缩放/alpha/UV 变化。

因此如果资源规划上要取舍：

- 整脸模式可以重点做多帧表情；
- 分层模式可以重点做高质量状态资源 + 程序形变 + pupil UV。

## 十、结论

当前 `CharacterStudio` 的“程序控制表情动画”已经不是单一的“表情枚举切图”，而是分成了几条清晰的已实现链路：

1. **整脸表情链**
   - 负责 FullFace / Head 贴图切换；
   - 支持方向变体与多帧整脸动画。

2. **分层状态链**
   - `Brow / Eye / UpperLid / LowerLid / Mouth / Emotion Overlay`
   - 由 `MouthState / LidState / BrowState / EmotionOverlayState` 驱动；
   - 同时支持状态资源切换与程序形变。

3. **眼方向链**
   - 旧系统：额外眼方向覆盖层；
   - 新主线：layered pupil 的 UV 偏移。

4. **左右独立基础**
   - `Brow / Eye / Pupil / UpperLid / LowerLid`
   - 已有左右独立资源、节点、图层基础；
   - 但运行时左右独立状态 API 还没有完全展开。

如果一句话总结当前实现状态：

> **现在已经能明确回答“哪个部件可以怎么动”，而且 layered face 的眉、眼睑、瞳孔、嘴、情绪层都已经有了真实运行时驱动；只是左右独立资源基础已经到位，左右独立状态控制还处在下一阶段。**
