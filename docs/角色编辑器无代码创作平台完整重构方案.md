# CharacterStudio 角色编辑器无代码创作平台完整重构方案

## 1. 文档目的

本文档用于为 CharacterStudio 的角色编辑器提供一套**完整、可落地、可分阶段实施**的重构方案。

目标不是继续围绕单点问题追加补丁，而是将当前编辑器从“可编辑皮肤与图层的工具集合”，升级为一个**以角色设计文档为核心、以无代码工作流为中心、以统一应用管线为底座**的角色创作平台。

本文档重点解决以下问题：

- 图层替换、挂载、隐藏等能力目前语义混杂，容易依赖运行时补救逻辑。
- 编辑器草稿、运行时皮肤、默认种族绑定、导出产物共用一套数据，导致职责边界不清。
- 预览、应用到 Pawn、应用到种族、导出发布虽然表面可用，但本质上仍是多条链路并行演化。
- 当前系统适合作为功能型编辑器，但离“稳定的无代码角色创作平台”仍有架构差距。

---

## 2. 平台定位

CharacterStudio 的角色编辑器核心定位应明确为：

> 一个面向 RimWorld 的无代码角色创作平台。

这意味着系统的核心目标不应只围绕图层替换，而应覆盖以下完整能力：

1. **原创角色创作**：从基础外观、图层、表情、武器到技能与属性的完整构建。
2. **种族兼容修正**：通过可视化方式对特定种族进行局部外观修补与覆盖。
3. **模板化复用**：将角色设计作为可复用模板应用到指定种族、指定 Pawn 或指定绑定入口。
4. **运行时稳定应用**：预览、保存、应用、默认种族绑定、导出使用统一规则。
5. **无代码发布**：在不编写 XML Patch 或 Harmony 代码的前提下完成可分发的内容产出。

因此，本方案以“平台化”而非“补丁式修复”为指导原则。

---

## 3. 当前架构现状与核心问题

### 3.1 当前主要模块概览

当前项目中，角色编辑器相关职责大致分布如下：

- 编辑器 UI 主体：`Source/CharacterStudio/UI/Dialog_SkinEditor*.cs`
- 运行时皮肤定义：`Source/CharacterStudio/Core/PawnSkinDef.cs`
- 运行时应用入口：`Source/CharacterStudio/Core/PawnSkinRuntimeUtility.cs`
- 默认种族绑定：`Source/CharacterStudio/Core/PawnSkinBootstrapComponent.cs`
- 渲染树注入与隐藏：
  - `Source/CharacterStudio/Rendering/Patch_PawnRenderTree.cs`
  - `Source/CharacterStudio/Rendering/Patch_PawnRenderTree.Injection.cs`
  - `Source/CharacterStudio/Rendering/Patch_PawnRenderTree.Hiding.cs`
  - `Source/CharacterStudio/Rendering/Patch_PawnRenderTree.NodeSearch.cs`
- 预览人偶：`Source/CharacterStudio/UI/MannequinManager.cs`
- 保存与导出：
  - `Source/CharacterStudio/Exporter/SkinSaver.cs`
  - `Source/CharacterStudio/Exporter/ModBuilder.cs`
  - `Source/CharacterStudio/Exporter/ModExportXmlWriter.cs`

### 3.2 核心结构性问题

#### 问题 A：编辑数据与运行时数据耦合过深

当前 `PawnSkinDef` 同时承担以下职责：

- 编辑器工作草稿
- 运行时实际生效 Def
- 默认种族绑定载体
- 导出源数据
- 种族修正的局部规则承载体

这会导致：

- 编辑器功能每扩展一次，就更依赖在 `PawnSkinDef` 上继续堆字段。
- 编辑器工作流无法保存更多“仅编辑期有效”的上下文。
- UI 行为和运行时行为之间缺少明确的编译边界。

#### 问题 B：语义模型不完整

当前很多操作是通过多个字段组合隐式表达的，例如：

- “替换某个原节点”并不是显式语义，而是：
  - 将旧节点路径加入 `hiddenPaths`
  - 再增加一条 `PawnLayerConfig`
- “修正某层偏移”与“新增附加图层”本质不同，但当前都落在图层配置中处理。

这使系统无法准确区分：

- Attach（附加）
- Replace（替换）
- Override（覆写）
- Hide（隐藏）

#### 问题 C：目标定位策略脆弱

当前运行时挂载目标节点主要依赖：

1. `anchorPath`
2. 回退到 `anchorTag`
3. 再回退到智能推断的 Head / Body

这导致：

- 上游种族结构变化时容易失效。
- 失效后经常不是显式失败，而是静默错挂。
- 用户看到的症状往往是“偏移异常、层级错误、局部像没渲染”。

#### 问题 D：多条应用链路并行存在

当前至少存在以下应用路径：

- 预览人偶应用
- 应用到目标 Pawn
- 保存后自动应用到目标 Pawn
- 保存后默认种族自动绑定
- 导出时重新组织数据

虽然底层已有一定共用，但上层规则仍分散，缺乏统一的“应用计划”。

#### 问题 E：编辑器更像功能集合，而非平台工作流

当前 UI 能力很多，但仍主要以：

- 图层树
- 节点树
- 属性面板
- 面部系统
- 武器系统
- 保存/应用/导出

这种“工具集合”形式存在。

它适合熟悉内部模型的开发者或高级用户，但不完全符合“无代码创作平台”的用户心智模型。

---

## 4. 重构总目标

本方案建议将系统升级为：

> 设计文档驱动的无代码角色创作平台。

总体目标如下：

### 4.1 架构目标

1. **编辑期数据与运行时数据解耦**
2. **所有创作行为都有明确语义模型**
3. **预览、应用、绑定、导出走统一执行计划**
4. **节点定位失败默认显式失败，不允许静默错误扩散**
5. **UI 以创作工作流为主，而不是以底层字段为主**

### 4.2 产品目标

1. 普通用户能够在无代码前提下完成角色创作与种族修正。
2. 高级用户能够构建复杂的视觉、技能、武器与属性组合。
3. 所有应用方式尽可能稳定、一致、可预测。
4. 导出结果清晰，运行时行为与编辑期预览保持高一致性。

---

## 5. 整体分层架构方案

建议将系统重构为 5 层：

1. **设计文档层（Design Document Layer）**
2. **规则编译层（Compilation Layer）**
3. **应用计划层（Application Plan Layer）**
4. **运行时执行层（Runtime Execution Layer）**
5. **无代码工作流层（Workflow/UI Layer）**

下面逐层说明。

---

# 模块一：设计文档层

## 6. 模块定位

设计文档层用于承载**编辑器内部的完整创作状态**，不直接等同于运行时 Def。

其职责是：

- 保存编辑器草稿
- 保存工作流上下文
- 保存节点引用与替换语义
- 保存尚未编译为运行时 Def 的设计信息
- 作为 UI 的唯一编辑对象

## 7. 建议新增核心模型

建议引入编辑器专用模型，例如：

- `CharacterDesignDocument`
- `CharacterVisualDocument`
- `CharacterPublishProfile`

推荐采用如下结构：

### 7.1 `CharacterDesignDocument`

作为顶层编辑文档，包含：

- 元数据
  - designId
  - title
  - description
  - author
  - version
- 预览上下文
  - preferredPreviewRace
  - preferredPreviewPawn
  - importedSources
- 可视设计部分
  - base appearance
  - layers
  - node operations
  - face config
  - weapon render config
- 角色能力部分
  - abilities
  - ability hotkeys
  - attributes
- 应用目标部分
  - targetRaces
  - default binding strategy
  - publish options
- 编辑器状态部分
  - undo/redo snapshots
  - dirty state
  - validation cache

### 7.2 `CharacterNodeReference`

用于稳定描述一个节点，而不是只存 `anchorPath`。

建议字段：

- exactNodePath
- terminalTag
- texPathHint
- workerClassHint
- siblingIndexHint
- sourceRaceDefName
- sourceTreeSignature

### 7.3 `CharacterNodeOperation`

显式表示对节点的操作，替代目前隐式组合。

操作类型建议为：

- `HideNodeOperation`
- `ReplaceNodeOperation`
- `OverrideNodeOperation`
- `AttachLayerOperation`

### 7.4 `CharacterPublishProfile`

保存发布层策略：

- applyToTargetRaces
- defaultRacePriority
- export mode
- dependency strategy
- runtime compatibility mode

## 8. 这一层的收益

- 编辑器不再直接篡改运行时 Def。
- 可以保存更多仅编辑器需要的信息。
- 替换、覆写、附加等语义终于有正式模型。
- 未来可以支持“项目文件”“草稿文件”“发布文件”三种不同输出。

---

# 模块二：规则编译层

## 9. 模块定位

规则编译层负责把 `CharacterDesignDocument` 转换成：

- 运行时可执行的视觉应用规则
- 可保存的 `PawnSkinDef`
- 可导出的发布内容

它是编辑器与运行时之间最关键的边界。

## 10. 建议新增核心组件

### 10.1 `CharacterDesignCompiler`

负责把设计文档编译成标准中间结果。

输出可包含：

- `CompiledCharacterVisual`
- `CompiledCharacterBindings`
- `CompiledCharacterExportPayload`

### 10.2 `CharacterVisualCompiler`

专门处理视觉部分，负责将：

- base appearance
- layers
- node operations
- face config
- weapon render config

编译为统一视觉规则集。

### 10.3 `CharacterPublishCompiler`

负责将发布配置编译为：

- 运行时 `PawnSkinDef`
- 默认种族绑定规则
- 导出清单

## 11. 编译原则

### 原则 A：运行时 Def 是“编译产物”，不是“编辑源文件”

`PawnSkinDef` 应仅承担：

- 运行时应用所需字段
- 导出所需最小字段

而不是完整编辑期文档。

### 原则 B：旧数据兼容读取，新数据优先走新模型

需要支持旧版本 `PawnSkinDef` 导入为设计文档，以确保迁移成本可控。

### 原则 C：编译期完成尽可能多的验证与降级

例如：

- 目标种族是否合法
- 图层纹理是否为空
- 节点引用是否存在足够的匹配信息
- 发布配置是否完整

---

# 模块三：应用计划层

## 12. 模块定位

应用计划层是整个系统的执行核心。

无论是：

- 预览
- 应用到 Pawn
- 默认种族自动绑定
- 导出前一致性校验

都应该先生成统一的 `CharacterApplicationPlan`，再由执行器提交。

## 13. 建议新增核心模型

### 13.1 `CharacterApplicationPlan`

建议包含：

- target pawn / target race info
- visual node resolution results
- nodes to restore
- nodes to hide
- nodes to replace
- layers to attach
- node overrides
- face runtime updates
- weapon render updates
- ability grants / revokes
- render rebuild strategy
- validation messages

### 13.2 `CharacterApplicationPlanBuilder`

负责基于：

- 设计文档
- 编译后视觉规则
- 当前 Pawn / 当前预览对象
- 当前种族渲染树快照

构建计划。

### 13.3 `CharacterApplicationValidator`

在计划提交前完成验证，给出状态：

- ExactMatch
- CompatibleMatch
- FallbackMatch
- NotMatched
- InvalidConfiguration

## 14. 为什么这一层重要

当前系统的问题之一，是不同入口自己拼装刷新逻辑。

如果统一为 `ApplicationPlan`：

- 所有入口都复用同一规则
- 所有问题都可以在“构建计划”阶段暴露
- 预览与运行时行为差异会显著减少

---

# 模块四：运行时执行层

## 15. 模块定位

运行时执行层负责对计划进行原子化提交。

这层应取代当前“边恢复、边删、边注入、边隐藏”的散乱过程，变为：

1. 读取计划
2. 分阶段提交
3. 完成渲染刷新

## 16. 建议新增核心组件

### 16.1 `CharacterApplicationExecutor`

唯一负责计划提交。

### 16.2 `RenderPlanExecutor`

专注处理渲染树层面的提交，执行顺序建议固定为：

1. Restore previous hidden state for current tree
2. Remove existing custom nodes
3. Resolve target nodes
4. Apply node hides
5. Apply node overrides
6. Apply node replacements
7. Attach additional nodes
8. Rebuild ancestors
9. Set dirty / refresh graphics

### 16.3 `NodeResolverService`

用于统一节点解析策略，不允许逻辑散落在多个文件。

## 17. 节点定位重构建议

当前节点定位不应继续只依赖：

- `anchorPath`
- `anchorTag`

建议统一为“节点引用解析策略”：

### 解析优先级建议

1. exact path
2. tag + texPath + sibling index
3. tag + worker class + terminal path hint
4. tag only

但必须明确：

- 对 Replace / Override 这类严格操作，未命中应直接失败。
- 对 Attach 这类附加操作，才允许有限回退。

## 18. 严格模式与宽松模式

建议显式区分：

### 18.1 Strict Replace Mode

- 找不到目标节点 => 失败
- 禁止挂到 root
- 记录错误，不提交部分结果

### 18.2 Attach Mode

- 允许部分 fallback
- 但必须标记匹配级别
- 在 UI 中反馈“非精确挂载”状态

## 19. 执行层收益

- 从根本上减少静默错误。
- 让“替换失败”变成明确状态，而不是错位渲染。
- 预览与实际应用共享同一执行底座。

---

# 模块五：无代码工作流层

## 20. 模块定位

无代码工作流层是面向用户的前台能力组织方式。

当前角色编辑器功能很多，但用户感知仍偏“功能列表”。

建议将编辑器改造为围绕工作流组织的创作平台。

## 21. 建议划分四大创作模式

### 21.1 模式 A：原创角色创作

面向从零构建完整角色。

功能入口：

- 基础外观
- 自定义图层
- 面部系统
- 武器视觉
- 技能系统
- 属性设定
- 角色预览

### 21.2 模式 B：种族修正 / 兼容创作

面向“无代码修模组外观问题”。

功能入口：

- 导入目标种族渲染树
- 节点选择
- 隐藏节点
- 替换节点
- 局部偏移与层级修正
- 目标种族绑定
- 替换状态验证

### 21.3 模式 C：模板化设计

面向“把设计复用到某类角色”。

功能入口：

- 目标种族策略
- 默认种族绑定
- 优先级配置
- 多模板切换
- 快速套用

### 21.4 模式 D：发布与导出

面向最终产物生成。

功能入口：

- 保存到本地设计库
- 编译为运行时皮肤
- 导出 Cosmetic Pack
- 导出 PluginOnly
- 导出 FullUnit
- 发布验证报告

## 22. 建议新增“向导式”子工作流

### 22.1 节点替换向导

流程：

1. 选中目标节点
2. 选择替换类型（替换 / 覆写 / 附加）
3. 自动记录节点引用
4. 自动生成对应规则
5. 自动预览命中状态
6. 自动提示风险

### 22.2 种族修正向导

流程：

1. 选择目标种族
2. 导入结构
3. 识别需修节点
4. 生成修正规则
5. 锁定目标种族
6. 保存为兼容设计

### 22.3 发布向导

流程：

1. 运行完整验证
2. 列出缺失资源 / 风险匹配
3. 选择导出模式
4. 生成产物
5. 输出发布说明

## 23. 工作流层收益

- 普通用户不必理解底层数据结构。
- 高级功能不再通过“知道哪个字段该怎么配”才能使用。
- 平台产品感显著增强。

---

# 模块六：验证与质量保障模块

## 24. 模块定位

作为无代码平台，系统必须在“保存前、应用前、导出前”提供稳定验证。

不应继续依赖运行时日志作为主要反馈手段。

## 25. 建议新增验证器

### 25.1 `CharacterDesignValidator`

校验设计文档完整性：

- defName / title / label
- 纹理路径合法性
- 技能引用合法性
- 表情配置完整性

### 25.2 `CharacterNodeRuleValidator`

校验节点规则：

- 是否存在目标引用
- 是否为严格替换但无可解析引用
- 是否可能命中多个节点
- 是否发生危险 fallback

### 25.3 `CharacterPublishValidator`

校验发布配置：

- targetRaces 是否为空或错误
- 默认种族绑定优先级是否冲突
- 导出模式是否缺少必要资源
- 依赖是否完整

## 26. 输出形式建议

每次验证应输出结构化结果：

- Error
- Warning
- Info
- Exact Match
- Fallback Match
- Compatibility Risk

并在编辑器 UI 中可视化展示。

---

# 模块七：数据迁移与兼容模块

## 27. 模块定位

由于现有用户已经在使用 `PawnSkinDef` 工作流，因此必须提供平滑迁移。

## 28. 迁移策略

### 28.1 旧皮肤导入为设计文档

将现有 `PawnSkinDef` 自动转换为：

- 基础外观
- 图层配置
- 目标种族配置
- 默认种族策略
- 面部与武器配置

### 28.2 旧式字段兼容

在短中期内继续兼容：

- `layers`
- `hiddenPaths`
- `hiddenTags`
- `targetRaces`

但仅作为旧数据输入，不再作为未来核心设计模型。

### 28.3 双轨过渡

过渡期可采用：

- 编辑器内部使用 `CharacterDesignDocument`
- 发布时仍编译到 `PawnSkinDef`

这样能避免一次性推翻现有运行时生态。

---

# 模块八：预览系统重构模块

## 29. 模块定位

预览必须与实际应用尽量一致，否则无代码平台会失去可信度。

## 30. 目标

将当前 `MannequinManager` 从“预览专用控制器”升级为“应用计划的轻量执行前端”。

## 31. 建议

### 31.1 预览也走 `ApplicationPlan`

不要让预览与正式应用是两套逻辑。

### 31.2 引入预览状态面板

显示：

- 当前种族
- 当前匹配节点数
- 替换规则命中状态
- fallback 命中情况
- 未命中节点

### 31.3 引入预览一致性检查

保存前运行“预览结果是否与运行时执行计划一致”的检查。

---

# 模块九：导出与发布模块

## 32. 模块定位

导出模块不应只是“把现有 Def 写出去”，而应成为**设计文档发布器**。

## 33. 建议

### 33.1 导出来源统一为设计文档

导出器应基于编译后的发布结果，而不是直接从运行时 Def 拼装。

### 33.2 增加导出报告

导出后生成报告，内容包括：

- 导出模式
- 目标种族
- 默认绑定策略
- 规则命中级别
- 资源依赖来源
- 已知兼容性风险

### 33.3 增加“兼容性级别”标记

例如：

- Stable for Fixed Race Version
- Structure Sensitive
- Preview Verified Only
- Strict Replace Validated

这能帮助用户理解产物的稳定性边界。

# 模块十：日志与可观测性模块

既然平台目标是无代码，就不能把调试完全依赖开发日志。

## 38. 关键设计原则

后续所有实现必须坚持以下原则：

### 原则 1：编辑数据不直接等于运行时数据

编辑器内部可以复杂，但运行时应用模型必须清晰、稳定、可编译。

### 原则 2：替换不是附加

替换、覆写、隐藏、附加必须有不同语义模型，不能继续靠组合字段模拟。

### 原则 3：定位失败要显式失败

严格替换不能静默挂 root，不能依赖隐式回退掩盖错误。

### 原则 4：所有应用路径共用一个执行计划

预览、应用到 Pawn、应用到种族、导出验证，必须共用同一逻辑核心。

### 原则 5：平台以工作流组织，而不是以字段组织

用户应看到“我要创作什么”，而不是“我要改哪个底层字段”。

### 原则 6：无代码不等于无验证

越是无代码平台，越需要更强的规则验证与状态反馈。

完成本方案后，CharacterStudio 将获得以下收益：

### 对用户

## 40. 最终结论

如果 CharacterStudio 的目标只是“继续修一个皮肤编辑器”，那么现有架构加补丁仍能勉强前进。

但如果目标是：

> 一个稳定的、长期可演进的、以无代码创作为核心的角色创作平台

那么必须进行本方案所描述的架构升级。

最根本的转变只有一句话：

> **从“直接编辑运行时皮肤定义”转向“编辑角色设计文档，并通过统一编译与应用计划驱动预览、应用与发布”。**

这不是局部修复，而是平台化升级的基础。
