### 1.2 整体架构评价

**优点：**
- 🟢 **模块化清晰**：通过 `partial class` 将 3955 行的 Worker 和 577+1212+373+265 行的 Patch 拆分为多个职责文件，可维护性良好
- 🟢 **线程安全设计**：[`RuntimeAssetLoader`](Source/CharacterStudio/Rendering/RuntimeAssetLoader.cs:49) 使用双锁分离（`textureCacheLock` / `materialCacheLock`），非主线程纹理加载有完善的排队机制
- 🟢 **渐进式隐藏策略**：使用 [`BaseContent.ClearMat`](Source/CharacterStudio/Rendering/Patch_PawnRenderTree.cs:252) 而非空材质，避免截断子节点渲染链
- 🟢 **已完成的 P0-P4 优化**：缓存扩容（512/1024）、文件修改检查节流、反向索引、边缘修正去重、Shader.PropertyToID 缓存

**缺点：**
- 🔴 **巨型 partial class 反模式**：[`PawnRenderNodeWorker_CustomLayer`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:16) 合计 3955 行 + 5 个 partial 文件共约 6300 行，职责过重
- 🔴 **大量代码重复**：Face.cs 与主文件包含同名方法的完整副本（见下文详细分析）
- 🟡 **反射依赖过重**：[`GetGraphicsField()`](Source/CharacterStudio/Rendering/Patch_PawnRenderTree.cs:53) 通过字段名猜测 + 类型探测反射访问内部字段

## 二、关键代码质量问题

### 2.1 🔴 严重代码重复：Face.cs 与主文件完全重复

[`PawnRenderNodeWorker_CustomLayer.Face.cs`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.Face.cs:1) 与 [`PawnRenderNodeWorker_CustomLayer.cs`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:838) 包含**完全相同**的方法实现：

| `ResetProgrammaticFaceTransform()` | 32-38 | 857-863 |
| `HideProgrammaticFacePart()` | 40-46 | 865-871 |

| `GetMouthMotionConfig()` | 227-233 | 1565-1571 |
| `GetEmotionOverlayMotionConfig()` | 235-242 | 1573-1579 |

**编译器会选择哪个？** C# partial class 中不允许同名方法重复定义——如果编译通过，说明其中一个文件的方法被条件编译或预处理器排除了，或者两个文件中的方法签名有微小差异。但从阅读的代码来看，**这些方法完全相同**，这意味着：
1. 编译器可能因为完全一致而产生歧义错误
2. 或者 Face.cs 中的版本实际上被某种机制覆盖了主文件版本

**建议**：删除主文件中的重复方法，保留 Face.cs 作为面部变换的唯一来源。

## 五、新发现的优化建议

### 5.1 🔴 P-NEW-1：消除残留 `TryGetComp` 调用（预估收益：-30-50μs/帧）

**问题**：第 2.4 节所列的 15+ 处直接 `TryGetComp` 调用完全可以使用已有的 [`GetCachedSkinComp()`](Source/CharacterStudio/Rendering/PawnRenderNode_Custom.cs:20)。

**方案**：在所有面部变换方法的入口处，将 `pawn.TryGetComp<CompPawnSkin>()` 替换为 `customNode.GetCachedSkinComp()`。对于非 `PawnRenderNode_Custom` 节点的路径（如 `OffsetFor` 的 line 185），保留 `TryGetComp` 作为 fallback。

**优先级**：高。改动面广但每处改动简单（纯替换），无行为变化。

### 5.2 🔴 P-NEW-2：消除代码重复（消灭编译风险 + 降低维护成本）

**问题**：Face.cs、Variant.cs、Overlay.cs 与主文件的大量方法重复。

**方案**：
1. 检查编译是否报错——如果不报错，说明存在隐式覆盖或 IDE 误导
2. 保留 partial 文件中的版本，删除主文件中的重复副本
3. 为每个 partial 文件添加明确的职责注释

**优先级**：高。代码重复是 bug 温床。

### 5.3 🟡 P-NEW-3：`GetGraphic()` 路径解析减少 List 分配（预估收益：-20-30μs/帧）

**问题**：[`GetLogicalSuffixCandidates()`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:2685) 和 [`BuildVariantPrefixes()`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:2811) 每次调用都分配新的 `List<string>`。每帧 150 个节点 × 每节点 2 个 List = 300 次 List 分配。

**方案**：
- 使用 `stackalloc` 或预分配的 `ThreadLocal<List<string>>` 缓冲区
- 或在 `PawnRenderNode_Custom` 上缓存上一帧的变体路径解析结果（路径依赖表情状态，可按 tick 节流）

### 5.4 🟡 P-NEW-4：`TryGetFollowSourceTransform()` BFS 替换为索引查找

**问题**：[`TryGetFollowSourceTransform()`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:2251) 每次调用遍历整棵渲染树。

**方案**：在注入面部节点时，在 `PawnRenderNode_Custom` 上维护一个 `Dictionary<LayeredFacePartType, PawnRenderNode_Custom>` 索引，查找从 O(N) 降为 O(1)。

### 5.5 🟡 P-NEW-5：`FindNodeByExactTag()` 使用已有缓存字段

**问题**：[`FindNodeByExactTag()`](Source/CharacterStudio/Rendering/Patch_PawnRenderTree.NodeSearch.cs:192) 每次调用 `AccessTools.Field()` 反射获取 `nodesByTag`。

**方案**：改用主文件已缓存的 [`nodesByTagField_Cached`](Source/CharacterStudio/Rendering/Patch_PawnRenderTree.cs:40)。

### 5.6 🟡 P-NEW-6：`ShouldSuppressEmotionOverlayRendering()` LINQ 开销

**问题**：主文件中的 [`ShouldSuppressEmotionOverlayRendering()`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:1981) 在行 1956 使用 `FirstOrDefault` + lambda + `Any()`，每帧对 Overlay 节点调用。

**方案**：将 `emotionOverlayRules` 的查找预编译为 Dictionary，避免每帧遍历。

### 5.7 🟢 P-NEW-7：`ResolveExistingTexturePathCore()` 目录扫描延迟

**问题**：[`ResolveExistingTexturePathCore()`](Source/CharacterStudio/Rendering/RuntimeAssetLoader.cs:297) 在路径不匹配时执行 `Directory.GetFiles()` + LINQ 链。虽然有 `resolvedPathCache` 缓存结果，但首次访问仍有高延迟。

**方案**：已有 P7 缓存。建议进一步在 `PrewarmExternalSkinAssets()` 中预热缓存，避免渲染帧首次命中。

### 5.8 🟢 P-NEW-8：动画计算可按 tick 节流

**问题**：[`CalculateAnimation()`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:480) 的 Swing/IdleSway/Breathe 类型使用 `Time.realtimeSinceStartup` 进行正弦波计算，实际上同一 tick 内结果不变。

**方案**：参照 `EnsureProgrammaticFaceStateUpdated()` 的 tick 节流模式，添加 `lastAnimationTick` 检查。

## 六、架构层面建议

### 6.1 Worker 类拆分

当前 [`PawnRenderNodeWorker_CustomLayer`](Source/CharacterStudio/Rendering/PawnRenderNodeWorker_CustomLayer.cs:16) 承担了以下所有职责：

1. **渲染管线接口**（CanDrawNow/GetGraphic/OffsetFor/ScaleFor/RotationFor/LayerFor/GetMaterialPropertyBlock）
2. **图层动画引擎**（Twitch/Swing/IdleSway/Breathe/Spin/Brownian）
3. **程序化面部变换**（眉/眼/瞳孔/上下眼睑/嘴/表情覆盖层）
4. **变体纹理解析**（表情后缀/方向后缀/帧序列/通道状态）
5. **设备触发动画**（部署/保持/返回三段动画）
6. **Overlay 语义匹配**（情感覆盖层显示/隐藏逻辑）

**建议**：将 2-6 拆分为独立的服务类，Worker 仅作为薄代理层调用各服务：

```
PawnRenderNodeWorker_CustomLayer (渲染接口)
├── LayerAnimationService (图层动画计算)
├── FaceTransformService (面部程序化变换) ← 已有 FaceTransformEvaluator 雏形
├── VariantResolver (变体纹理解析)
├── TriggeredAnimationService (设备触发动画)
└── OverlaySemanticService (覆盖层语义判断)
```

### 6.2 节点数据模型增强

[`PawnRenderNode_Custom`](Source/CharacterStudio/Rendering/PawnRenderNode_Custom.cs:11) 当前有 **28 个公共字段**用于存储各种动画状态。建议：

1. 将动画状态封装为 `struct LayerAnimationState`
2. 将程序化面部状态封装为 `struct ProgrammaticFaceState`
3. 将外部图层动画状态封装为 `struct ExternalLayerAnimationState`

这样既保持内存连续性（struct），又提高代码可读性。

### 6.3 隐藏策略的 `ConditionalWeakTable` 替代方案

[`Patch_PawnRenderTree.Hiding.cs`](Source/CharacterStudio/Rendering/Patch_PawnRenderTree.Hiding.cs:18) 使用 `ConditionalWeakTable` 存储隐藏状态。虽然不会造成内存泄漏，但 `ConditionalWeakTable` 在高频查找时性能不如 `Dictionary`（内部使用弱引用+GC handle）。

考虑到 `GetFinalizedMaterial_Postfix` 每帧对所有节点触发，`hiddenNodesByTree.TryGetValue()` 的调用频率极高（500-1000/帧）。建议评估替换为 `Dictionary` + 显式清理的方案。
