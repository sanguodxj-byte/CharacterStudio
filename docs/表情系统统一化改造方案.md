# CharacterStudio 表情系统统一化改造方案

## 1. 背景与目标

本文档整理当前对 CharacterStudio、Facial Animation 本体模组（`1635901197`）以及 Koelime 动态表情适配包（`3586343826`）的分析结果，并提出一套适合 CharacterStudio 后续实现的表情系统修改方案。

### 1.1 当前问题

CharacterStudio 当前与表情相关的逻辑主要分散在以下几套结构中：

- `CompPawnSkin`
  - 负责计算当前表情状态、眨眼、眼睛方向、预览覆盖等
- `PawnFaceConfig`
  - 负责整脸表情配置、帧表情、LayeredDynamic 数据结构
- `PawnRenderNodeWorker_FaceComponent`
  - 负责整脸/整头贴图切换
- `PawnRenderNodeWorker_EyeDirection`
  - 负责眼睛方向覆盖层
- `PawnLayerConfig` + `PawnRenderNodeWorker_CustomLayer`
  - 负责普通图层与通用图层动画

这会带来几个问题：

1. **普通图层与表情图层是两套平行系统**
2. **AI 整图拆件资源接入不自然**
3. **后续扩展嘴型、眼睑、腮红、泪水等时结构偏重**
4. **编辑器 authoring 成本偏高**
5. **表情逻辑与图层逻辑没有真正统一**

## 2. 外部参考结论

## 2.1 Facial Animation 本体（`1635901197`）

通过对 `D:\steam\steamapps\workshop\content\294100\1635901197\1.6` 的检查，可以确认：

### 结构特点

- `Assemblies/FacialAnimation.dll`
- `Defs/AnimationDefs/...`
- `Defs/FaceShapeDefs/...`
- `Defs/FaceTypeDefs/Human/...`
- `Textures/Things/Pawn/Humanlike/...`

### 逻辑特点

其核心是一个 **数据驱动的分件式面部动画框架**：

- 默认脸：`normal`
- 周期动作：`blink`、`eyeMoving`
- Job 驱动动作：`Ingest`、`AttackMelee`、`LayDown` 等
- 每个动画由 `FaceAnimationDef` 定义
- 一个动画可以同时控制多个通道：
  - `headShapeDef`
  - `browShapeDef`
  - `lidShapeDef`
  - `mouthShapeDef`
  - `emotionShapeDef`
  - `eyeballOffset`
  - `headOffset`
  - `browOffset`
  - `lidOffset`

### 借鉴点

最值得借鉴的不是其完整重型结构，而是：

1. **状态 = 多通道组合**
2. **默认状态 + 高优先级小动画覆盖**
3. **小序列（sequence）而非纯静态切图**
4. **动画规则与资源解耦**

## 2.2 Koelime 动态表情适配包（`3586343826`）

通过对 `D:\steam\steamapps\workshop\content\294100\3586343826` 的检查，可以确认：

### 它本质上是什么

它不是独立框架，而是 **基于 Facial Animation 的种族适配包**。

### 资源组织方式

它将面部拆成这些主要部件：

- Head
- Brow
- Emotion
- Eye
- Lid
- LidOption
- Mouth
- Skin

其中核心表情相关的是：

- Head
- Brow
- Emotion
- Eye
- Lid
- LidOption
- Mouth

### 借鉴点

它说明：

1. 表情系统完全可以按“部件槽位”组织
2. 一个状态不需要对应一张整脸
3. 可以通过多个部件叠加组合出最终表情

## 2.3 对 CharacterStudio 的启示

### 不应照搬的部分

CharacterStudio 不应直接复制 Facial Animation 的完整设计：

- 多 ControllerComp
- 多 ShapeDef
- 多 TypeDef
- 重型 authoring 流程

原因：

- CharacterStudio 当前更偏图层系统
- 用户资源大量来自 AI 整图 + 局部重绘 + 抠图
- 资源生产方式并不适合传统精细分件 authoring

### 应吸收的核心思想

CharacterStudio 应吸收的是：

1. **状态按通道分解**
2. **图层按状态切换变体**
3. **允许轻量序列动画**
4. **默认态 + 覆盖态**
5. **让表情系统建立在统一图层系统之上**

---

## 3. 总体方案

## 3.1 方案名称

**Unified Layer Expression System**  
**统一图层表情系统**

## 3.2 核心思想

把表情系统改造成：

> 所有面部相关资源都视为普通图层，只是某些图层具有“按状态解析贴图变体”的能力。

也就是说：

- 身体层：普通图层
- 头发层：普通图层
- 嘴层：普通图层，但启用嘴型状态变体
- 眼睛层：普通图层，但启用眼神状态变体
- 眼睑层：普通图层，但启用 Blink/Sleep 状态变体
- 腮红层：普通图层，但启用情绪覆盖逻辑

最终目标：

- **普通层与表情层统一为同一套 Layer 系统**
- **表情不再是独立大系统，而是图层系统上的一种行为模式**

---

## 4. 设计原则

## 4.1 第一原则：统一数据模型

新系统下，优先使用 `PawnLayerConfig` 承载面部资源，而不是继续把表情主流程建立在 `PawnFaceConfig` 上。

## 4.2 第二原则：状态中心单一

`CompPawnSkin` 继续负责计算运行时状态，但不再直接主导复杂的贴图选择逻辑。

它只输出：

- 当前高层表情状态
- 当前眼睛方向
- 当前眨眼状态
- 当前序列 Tick
- 当前分通道状态

## 4.3 第三原则：解析逻辑集中

所有绝大部分图层的变体查找，应统一收敛到：

- `PawnRenderNodeWorker_CustomLayer`

## 4.4 第四原则：兼容旧资源

保留：

- `PawnFaceConfig`
- `PawnRenderNodeWorker_FaceComponent`
- `PawnRenderNodeWorker_EyeDirection`

但将它们逐步降级为：

- 兼容模式
- 旧资源模式
- 高级模式或 Legacy 模式

## 4.5 第五原则：优先适配 AI 资源工作流

资源输入方式应默认支持：

- 整图底图
- 同画布透明覆盖层
- 基于局部重绘提取的小部件
- 后缀命名驱动的轻量变体逻辑

---

## 5. 新架构概览

## 5.1 `CompPawnSkin`：状态中心

`CompPawnSkin` 仍然保留，但职责明确为：

### 负责：

- 当前 `ExpressionType`
- 当前 `EyeDirection`
- 当前是否处于 Blink
- 当前动画 Tick
- 预览覆盖状态
- 武器视觉状态等其他运行时状态

### 新增：

将高层语义状态进一步分解为更适合图层消费的通道状态，例如：

- `MouthState`
- `LidState`
- `BrowState`
- `EmotionOverlayState`
- `EyeState`

### 示例映射

- `Happy`
  - mouth = Smile
  - lid = Normal
  - brow = Normal
  - emotion = Happy
- `Sad`
  - mouth = Down
  - lid = Half
  - brow = Sad
  - emotion = Gloomy
- `Blink`
  - lid = Blink
- `Sleeping`
  - lid = Close
  - mouth = Sleep
- `Eating`
  - mouth = Open + Sequence
- `Lovin`
  - emotion = Lovin
  - mouth = Smile

---

## 5.2 `PawnLayerConfig`：统一图层承载体

### 当前职责

- 普通图层配置
- 偏移/缩放/旋转
- 动画参数

### 建议新增职责

让 `PawnLayerConfig` 支持“状态逻辑驱动的纹理变体”。

### 推荐新增字段

```csharp
public LayerRole role = LayerRole.Decoration;
public LayerVariantLogic variantLogic = LayerVariantLogic.None;
public string variantBaseName = "";
public bool useDirectionalSuffix = true;
public bool useExpressionSuffix = false;
public bool useEyeDirectionSuffix = false;
public bool useBlinkSuffix = false;
public bool useFrameSequence = false;
public bool hideWhenMissingVariant = false;
public EyeRenderMode eyeRenderMode = EyeRenderMode.TextureSwap;
public float eyeUvMoveRange = 0f;
public string[] visibleExpressions;
public string[] hiddenExpressions;
```

### 推荐新增枚举

```csharp
public enum LayerRole
{
    Base,
    Head,
    Brow,
    Eye,
    Lid,
    Mouth,
    Emotion,
    SkinMark,
    Decoration
}

public enum LayerVariantLogic
{
    None,
    ExpressionOnly,
    EyeDirectionOnly,
    BlinkOnly,
    ExpressionAndDirection,
    ChannelState,
    Sequence
}

public enum EyeRenderMode
{
    TextureSwap,
    UvOffset
}
```

---

## 5.3 `PawnRenderNodeWorker_CustomLayer`：统一解析入口

### 新职责

将绝大多数图层的贴图解析逻辑统一集中到这里。

### 目标

以后绝大多数层，包括：

- 嘴
- 眼
- 眼睑
- 腮红
- 泪水
- 眉毛
- 全脸覆盖层

都走 `CustomLayer`。

### 解析流程建议

1. 取基础 `texPath` 或 `variantBaseName`
2. 按朝向尝试方向后缀
3. 按通道状态尝试状态后缀
4. 按 Expression 尝试高层表达后缀
5. 按 EyeDirection 尝试眼神后缀
6. 按 Blink 尝试 Blink 后缀
7. 按帧序列尝试帧后缀
8. 执行回退链
9. 找不到时回退基础图或隐藏图层

## 10. 资源工作流建议

## 10.1 资源来源假设

系统默认要对以下工作流友好：

- AI 整图生成
- AI 局部重绘
- AI 抠图得到透明覆盖件

---

## 10.2 推荐资源形式

### 底图
- `body_base.png`

### 面部覆盖件
- `eye_Left.png`
- `eye_Right.png`
- `lid_Blink.png`
- `mouth_Smile.png`
- `mouth_Open.png`
- `emotion_Blush.png`

### 关键要求

- 与底图同画布尺寸
- 位置完全对齐
- 透明背景
- 默认 `offset = 0`

这样用户几乎不需要手工调坐标。

---

## 10.3 一层一职责

建议每个图层只负责一种表现：

- 嘴层只负责嘴
- 眼层只负责眼球/眼神
- 眼睑层只负责 Blink / Half / Close
- 情绪层只负责 blush / tear / gloom
- 眉毛层只负责眉型变化

避免一张图同时承担多个逻辑职责。

## 11. 实现阶段划分

## 11.1 Phase 1：MVP

### 目标
不破坏现有系统，尽快让新资源工作流跑通。

### 实现内容

- 给 `PawnLayerConfig` 增加角色/变体相关字段
- 扩展 `PawnRenderNodeWorker_CustomLayer.GetGraphic()`
- 增加通道状态解析函数
- 编辑器支持设置 layer role 与 preview state

### 优先支持的通道

- Mouth
- Lid
- Eye（TextureSwap）
- Emotion overlay
- Brow（可选）

### 本阶段不做

- UV 眼球偏移全面整合
- 删除旧 FaceConfig
- 高复杂度序列系统全面化

---

## 11.2 Phase 2：增强

### 目标
吸收 Facial Animation 的“微动作”优势。

### 实现内容

- 增加简单 sequence 支持
- 帧后缀解析
- Blink 使用短 sequence，而不是单张图硬切
- 简单 job 动画：
  - Eating → 嘴 open/normal 序列
  - Talking → 嘴 open/normal 序列
  - Sleep → lid close

---

## 11.3 Phase 3：统一收束

### 目标
将新资源主流程彻底统一到图层系统。

### 实现内容

- 眼睛方向逻辑并入 `CustomLayer`
- `FaceComponent` 转为纯 Legacy
- `PawnFaceConfig` 只保留兼容与旧数据支持
- 导出器优先导出统一图层表达

---

## 13. 为什么该方案比当前方案更适合 CharacterStudio

## 13.1 对资源生产更友好

- AI 资源直接适配
- 不需要重型 authoring
- 局部抠图可直接成为图层变体

## 13.2 对代码维护更友好

- 普通层与表情层统一
- 解析逻辑集中
- `CompPawnSkin` 职责清晰

## 13.3 对后续扩展更友好

- 可以先做轻量方案
- 再逐步加入 sequence、job 动画、UV eye shift
- 保留向更复杂系统靠近的空间

## 14. 最终结论

推荐 CharacterStudio 后续采用：

> **统一图层表情系统（Unified Layer Expression System）**

作为新的主工作流。

### 主工作流原则

- 普通图层与表情图层统一
- 由 `CompPawnSkin` 输出高层状态与通道状态
- 由 `PawnLayerConfig` 承载状态解析配置
- 由 `PawnRenderNodeWorker_CustomLayer` 统一负责纹理解析、后缀查找、回退链
- 旧 `PawnFaceConfig + FaceComponent` 保留为 Legacy 兼容模式

### 一句话总结

> **把表情系统从“独立的整脸配置系统”改成“图层系统上的状态解析能力”，让嘴、眼、眼睑、眉毛、情绪覆盖都成为普通图层，只是这些图层会根据运行时状态自动选择后缀纹理，并按统一回退链解析。**

---

## 15. 实施建议（简版）

如果后续按本文档开始实现，建议优先顺序如下：

1. 先改 `PawnLayerConfig`
2. 再改 `CompPawnSkin` 的通道状态输出
3. 再改 `PawnRenderNodeWorker_CustomLayer.GetGraphic()`
4. 再改编辑器预览与属性面板
5. 最后处理 Legacy 兼容与旧资源迁移

这样风险最低，收益最高。
