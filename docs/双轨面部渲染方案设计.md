# CharacterStudio 双轨面部渲染方案设计

## 1. 背景

当前 CharacterStudio 的面部系统主要基于分层贴图运行时组合：

- Base / Brow / Eye / UpperLid / LowerLid / Mouth / Overlay 多图层
- 运行时根据表情、朝向、状态选择纹理变体
- 通过 offset / scale / rotation / UV 偏移制造动态感

这套方案在编辑器友好性、资源可控性、导出工作流上有明显优势，但在 RimWorld 这种 **主线程极其宝贵的单线程游戏** 中，运行时仍然存在以下问题：

1. 多节点注入与维护成本较高
2. 纹理路径解析、fallback、变体查找带来 CPU 开销
3. 多层资源在世界视图下性价比不高
4. 后期大量 pawn 同屏时，主线程压力会快速放大

另一方面，NLFA 证明了把部分渲染复杂度推向 mesh / material / render texture / shader 管线是有效方向，但其后期性能与复杂度也说明：

- 不能简单照搬高质量 facial animation 路线
- 必须根据 RimWorld 的运行特点做 **强 LOD 与职责分离**

因此，需要一套新的双轨方案：

- **编辑态保留分层资源与编辑器优势**
- **运行时按使用场景切换为不同渲染轨道**
- **世界轨以低 CPU、低节点、低组织成本为优先**
- **肖像轨以表现力、局部动态、可视质量为优先**

---

## 2. 核心设计目标

## 2.1 总目标

建立一套 **分层编辑、编译输出、双轨渲染** 的面部系统，使 CharacterStudio 同时具备：

- 高可编辑性
- 高资源生产效率
- 后期可接受的运行时性能
- 近景较高的表情表现力

## 2.2 具体目标

### 编辑目标
- 继续使用分层资源作为作者输入格式
- 保留当前 Face Editor / Overlay 排序 / 局部替换工作流
- 尽量不要求画师直接处理 shader 参数格式

### 运行时目标
- 世界视图下最大限度减少：
  - 节点数量
  - 纹理路径查找
  - 变体 fallback
  - 每 tick 字符串处理
- 肖像与近景视图下允许更高质量表现
- 避免为每个 pawn 在世界视图中维持高复杂度 facial animation

### 架构目标
- 明确“编辑态数据”和“运行时数据”的边界
- 支持资源编译缓存
- 支持按轨道独立降级与调优
- 为后续 shader 参数化面部系统打基础

---

## 3. 非目标

以下内容不作为第一阶段目标：

1. 不直接复刻 NLFA 的完整 facial animation controller 架构
2. 不在第一阶段引入复杂 RenderTexture per-pawn 合成
3. 不在第一阶段强行把全部面部动画迁移为 shader 驱动
4. 不推翻当前 CharacterStudio 的编辑器与资源格式
5. 不以“一步到位替换所有现有渲染路径”为目标

---

## 4. 总体架构

## 4.1 双轨定义

### A. 世界轨（World Track）
面向：
- 地图常规视角
- 大量 pawn 同屏
- 性能优先场景

核心原则：
- 单节点或尽量少节点
- 尽量少材质切换
- 尽量少纹理变体查找
- 不做复杂高频 facial animation 控制
- 允许表现力降级，但要求稳定

建议形态：
- 轻量参数脸
- 优化版 FullFaceSwap
- 或“编译后 atlas + 少量参数”的单轨输出

### B. 肖像轨（Portrait Track）
面向：
- 角色肖像
- 选中单位
- 编辑器预览
- 高可视关注对象

核心原则：
- 允许更高质量
- 允许更多局部动态
- 允许有限的多层表现
- 优先还原角色设计与面部细节

建议形态：
- 增强版分层动态
- 参数化动态 + 局部 overlay
- 更完整的表情状态与动画支持

---

## 4.2 编辑态与运行态分离

### 编辑态（Authoring Format）
继续使用当前分层资源模型：

- Base
- Brow
- Eye
- Pupil
- UpperLid
- LowerLid
- Mouth
- Overlay（含 overlayId / overlayOrder）
- 各 expression / direction / anchorCorrection / offset 信息

### 运行态（Runtime Format）
不再直接依赖作者输入纹理路径做实时选择，而是编译成面向具体轨道的缓存格式。

运行态产物分为：

#### 世界轨运行时产物
- 编译后的基础脸输出路径
- 可选 atlas
- 可选 mask
- 少量参数映射
- 方向映射缓存
- expression 到轻量状态的映射表

#### 肖像轨运行时产物
- 更完整的层数据
- 更丰富的朝向/表情变体映射
- 局部动态参数
- overlay 编排结果
- 预解析路径缓存

---

## 5. 资源管线设计

## 5.1 源资源格式保留分层

画师继续按当前 CharacterStudio 方式制作：

- `Base`
- `Brow`
- `Eye`
- `UpperLid`
- `LowerLid`
- `Mouth`
- `Overlay_*`
- 可选 East / North 方向图
- 可选表情变体

原因：

1. 对美术直观
2. 可局部迭代
3. 易于编辑器操作
4. 易于导入导出
5. 能兼容当前生态

---

## 5.2 编译阶段新增运行时构建

需要新增一层 **Face Runtime Compiler**，将编辑态资源编译为运行时使用的结构。

建议输出两个目标：

### 世界轨编译目标
1. 预解析方向与表情可用性
2. 将高频使用资源整理为固定运行时引用
3. 为后续 atlas 化 / mask 化预留接口
4. 输出世界轨的轻量渲染描述

### 肖像轨编译目标
1. 预构建层排序结果
2. 缓存 expression -> part path 映射
3. 缓存 overlay 组与 draw order
4. 缓存 anchor / offset / correction

---

## 5.3 编译器的职责边界

编译器应负责：

- 资源存在性扫描
- 方向 fallback 规则固化
- 表情映射固化
- Overlay 排序固化
- 输出运行时索引

编译器不负责：

- 实时动画逻辑
- 每帧材质参数更新
- 游戏运行时状态决策

---

## 6. 世界轨设计

## 6.1 设计原则

世界轨是性能核心，目标不是“最精细”，而是：

- 足够像角色本人
- 能表达基础状态差异
- 不压主线程
- 可随着距离和场景降级

---

## 6.2 世界轨推荐渲染形态

第一阶段推荐采用：

### 方案 W1：轻量化单节点脸
- 尽量使用单节点输出
- 使用预编译后的基础脸纹理
- 支持极少量参数：
  - blink
  - eye look
  - mouth open
  - 简单 overlay 强度
- 不在世界轨维持完整分层多节点

### 方案 W2：保底 FullFaceSwap+
- 在远景和高负载场景直接退化为整脸切图
- 可选少量 shader 参数增强
- 强化缓存，尽可能无运行时路径查找

建议：
- 第一阶段落地 W2
- 第二阶段推进 W1

---

## 6.3 世界轨 LOD 策略

建议至少分为 3 档：

### LOD0：重点对象
适用：
- 玩家当前选中 pawn
- 鼠标 hover 高关注 pawn
- 屏幕中央关键角色

允许：
- 更完整表情
- 少量动态参数
- 更高更新频率

### LOD1：普通可见 pawn
适用：
- 绝大多数地图中的可见 pawn

仅保留：
- 基础表情状态
- 低频更新
- 少量动态

### LOD2：远景或高负载降级 pawn
适用：
- 屏幕边缘
- 远距离
- 大规模同屏
- 性能压力高时

退化为：
- 静态轻量脸
- 或基础 FullFaceSwap
- 或完全关闭复杂面部动态

---

## 6.4 世界轨更新频率

建议不要每帧更新，而采用事件 + 低频轮询：

- job / drafted / sleep / pain / mental state 变化时更新
- 普通表情参数每 10~30 tick 更新一次
- 远景可提高到 30~60 tick
- 屏幕外或不重要 pawn 冻结

---

## 7. 肖像轨设计

## 7.1 设计原则

肖像轨承担：
- 角色辨识度
- 面部表情表现力
- 作者设计还原度
- 编辑器预览一致性

肖像轨可以比世界轨复杂，但仍应避免无节制增长。

---

## 7.2 肖像轨建议能力

支持：

- 分层动态面部
- 更完整 overlay 表现
- 更丰富的 eye / lid / mouth 状态
- 更高更新频率
- 可选参数化补偿

不建议：

- 对所有 pawn 一视同仁启用最高级效果
- RenderTexture per-pawn 常驻
- 每 tick 重复做高成本路径解析

---

## 7.3 肖像轨的优化原则

即使是肖像轨，也应：

1. 缓存路径解析结果
2. 缓存表达式映射
3. 缓存可用方向资源
4. 将 overlay 排序结果预编译
5. 尽量把连续变化转为参数，而不是新增贴图状态

---

## 8. 参数化方向（未来演进）

长期目标不是继续增加更多分层节点，而是：

## 8.1 将分层资源逐步编译为参数脸输入
例如：

- 眉毛区域 mask
- 眼睛区域 mask
- 嘴巴区域 mask
- overlay mask
- atlas UV 信息
- shader 参数约定

## 8.2 运行时只传少量参数
例如：

- eyeOpen
- eyeLookX / eyeLookY
- mouthOpen
- mouthSmile
- browTension
- blushStrength
- sweatStrength
- tearStrength

## 8.3 shader 负责局部变化
逐步减少：
- 多节点
- 多路径
- 多材质
- 多 fallback

---

## 9. 数据结构建议

## 9.1 新增编译结果数据对象

建议新增：

### FaceRuntimeCompiledData
用于描述单个皮肤最终编译结果，包含：

- 世界轨描述
- 肖像轨描述
- 方向可用性
- expression 映射缓存
- overlay 编排缓存
- 资源版本号 / 构建标识

### FaceWorldTrackData
包含：

- world 主渲染路径
- 简化表情映射
- 可用方向缓存
- LOD 配置
- 基础参数开关

### FacePortraitTrackData
包含：

- portrait layer mapping
- overlay group layout
- expression part cache
- anchor correction cache
- 参数化支持标记

---

## 9.2 新增运行时状态对象

建议新增：

### FaceRuntimeState
用于描述游戏运行时当前状态，而不是资源内容：

- currentTrack
- currentLOD
- currentExpression
- eyeDirection
- mouthState
- overlay active flags
- nextUpdateTick
- dirty flags

将“资源”和“状态”严格分离。

---

## 10. 模块拆分建议

## 10.1 新模块：Runtime Compilation
建议放在：
- `Source/CharacterStudio/Rendering/RuntimeCompilation/`
或
- `Source/CharacterStudio/Core/RuntimeFace/`

建议职责：
- 源数据扫描
- 编译缓存构建
- 世界轨输出
- 肖像轨输出

## 10.2 新模块：Track Selection
职责：
- 判断当前 pawn 用世界轨还是肖像轨
- 根据场景与配置分配 LOD
- 控制更新频率

## 10.3 新模块：World Face Renderer
职责：
- 世界轨轻量绘制
- 强缓存
- 极少节点

## 10.4 新模块：Portrait Face Renderer
职责：
- 高质量预览
- 编辑器一致性
- 有限动态表现

---

## 11. 分阶段开发计划

## 阶段 1：建立双轨文档与数据边界
目标：
- 完成方案文档
- 明确编译态 / 运行态边界
- 设计新数据对象与接口

交付：
- 本文档
- 新增编译数据模型草案
- Track 选择接口草案

---

## 阶段 2：引入运行时编译缓存
目标：
- 将当前 FaceConfig 预编译为缓存结构
- 消除高频路径解析

交付：
- `FaceRuntimeCompiledData`
- 编译入口
- 缓存生命周期管理
- 基础调试输出

---

## 阶段 3：落地世界轨保底实现
目标：
- 先实现 FullFaceSwap+ 的世界轨
- 让世界渲染脱离当前高复杂度分层逻辑

交付：
- 世界轨渲染 worker
- Track 切换逻辑
- LOD 基础规则
- 配置开关

---

## 阶段 4：肖像轨接入编译缓存
目标：
- 保留现有效果，但减少运行时查找

交付：
- 肖像轨路径缓存
- overlay 编排缓存
- expression cache
- anchor cache

---

## 阶段 5：参数化世界轨
目标：
- 将世界轨进一步推进到单节点参数脸

交付：
- atlas/mask 原型
- shader 参数协议
- 世界轨参数下发逻辑

---

## 12. 第一阶段开发建议

文档完成后，建议立即开始以下开发顺序：

1. 新建双轨运行时数据结构
2. 新建 track / LOD 决策入口
3. 从当前 FaceConfig 生成基础编译缓存
4. 先让“世界轨 = 保底 FullFaceSwap+”
5. 再逐步把当前分层肖像轨接到缓存层上

原因：
- 风险最低
- 可逐步验证
- 能快速看到主线程减负收益
- 不破坏现有编辑器工作流

---

## 13. 最终结论

CharacterStudio 不应继续把“运行时分层渲染”作为唯一终局。

更合理的演进路线是：

- **保留分层资源与编辑器**
- **新增运行时编译层**
- **采用世界轨 / 肖像轨双轨渲染**
- **世界轨优先性能，肖像轨优先表现**
- **长期演进到参数化轻量运行时系统**

这样可以同时保留：

- 美术资源生产友好性
- 编辑器可编辑性
- 项目可控性
- 后期性能空间
