# CharacterStudio 表情模块优化方案

## 1. 背景

当前 CharacterStudio 的表情模块已经具备以下基础能力：

- 支持 **FullFaceSwap** 与 **LayeredDynamic** 两套工作流
- 支持整脸贴图切换、分层部件配置、Overlay 分组与自动导入
- 支持运行时表情推断、眼睛方向推断、Blink、预览覆盖
- 支持通过 `PawnRenderNodeWorker_FaceComponent` 与 `PawnRenderNodeWorker_CustomLayer` 完成运行时渲染

现阶段的主要问题不在“能不能用”，而在：

1. **运行时触发语义偏弱**，很多已导入资源不能被充分利用  
2. **表情动画刷新精度一般**，多帧动画切换时机仍可优化  
3. **编辑器预览更偏手工测试**，缺少一键验证典型情境的能力  
4. **眼睛方向回退逻辑较保守**，无目标时表现容易过于呆板

因此，本轮优化目标是将表情模块从“资源切换系统”进一步推进到“状态驱动的面部表现系统”。

---

## 2. 总体优化目标

### 2.1 功能目标
- 让更多 Overlay 资源在运行时真正自动触发
- 提高整脸帧动画切换的刷新准确性
- 增强编辑器中的表情验证能力
- 改善眼睛方向在无目标时的回退表现

### 2.2 实施原则
- 优先选择 **低风险、可回归验证、与现有架构兼容** 的改动
- 避免一次性推翻 `CompPawnSkin` / `PawnFaceConfig` 的现有数据结构
- 优先在运行时解析层和编辑器预览层补强能力
- 保持已有素材命名规则与自动导入工作流兼容

---

## 3. 优化项拆解

## 3.1 运行时 Emotion Overlay 触发增强

### 现状
当前 `EmotionOverlayState` 虽然定义了：

- `Blush`
- `Tear`
- `Gloomy`
- `Lovin`

但运行时自动解析主要集中在：

- `Lovin`
- `Gloomy`
- `None`

这会导致：

- `Blush`
- `Tear`
- `Sweat` 对应资源即使已导入，也难以自动出现

### 优化方向
在 `CompPawnSkin` 的运行时状态解析中补强自动触发逻辑，尽量利用已有状态源：

- `Lovin` → 维持 `Lovin`
- `Happy / Cheerful / SocialRelax` → 可触发 `Blush`
- `Hopeless / Pain / Shock / Dead` → 可触发 `Tear`
- `Scared / WaitCombat / AttackMelee / AttackRanged` → 可触发 `Sweat` 风格覆盖层

### 风险说明
当前 `EmotionOverlayState` 是单值状态，并非多 Overlay flag 叠加系统，因此本轮不直接改成多通道叠加，只做 **单状态自动触发增强**。

---

## 3.2 整脸帧动画刷新精度优化

### 现状
`CompPawnSkin.UpdateAnimatedExpressionFrame()` 当前主要按总周期边界触发刷新。  
这在简单动画下可用，但对多帧动画来说，不一定总能准确覆盖每次帧切换。

### 优化方向
改为：

- 记录当前表情动画对应的 **帧索引**
- 对比上一 Tick 的帧索引
- 仅在帧索引发生变化时请求渲染刷新

### 收益
- 刷新时机更准确
- 避免“只在整轮循环边界刷新”的问题
- 不需要改变既有数据结构

---

## 3.3 预览系统增加一键情境预设

### 现状
当前预览系统已支持分别覆盖：

- Expression
- EyeDirection
- Mouth
- Lid
- Brow
- Emotion Overlay

但主要还是手工逐项切换，验证效率较低。

### 优化方向
在编辑器预览中增加典型状态预设，例如：

- `Auto`
- `Combat`
- `Panic`
- `Tired`
- `Depressed`
- `Romance`
- `Downed`
- `Dead`

每个预设一次性写入一组预览覆盖，便于快速验证整套面部表现。

### 收益
- 创作者不需要逐个点选状态
- 能更快发现素材缺失和不协调问题
- 有利于分层面部工作流调试

---

## 3.4 眼睛方向回退逻辑优化

### 现状
当前眼睛方向优先使用 Job 目标格推断；当没有明确目标时，回退逻辑比较保守，容易长期停留在 `Center`。

### 优化方向
在无目标的情况下，补强朝向回退逻辑：

- 基于 `Pawn.Rotation` 提供更合理的视线方向默认值
- 保持死亡、倒地、睡眠等状态仍强制回中

### 收益
- 角色在无交互目标时不再总是“死盯中心”
- 与已有眼睛方向贴图系统兼容
- 风险较低

---

## 4. 本轮实施范围

本轮实际落地以 **低风险高收益** 为原则，优先实施以下四项：

1. **Emotion Overlay 自动触发增强**
2. **帧动画刷新改为帧索引变化驱动**
3. **预览系统增加典型情境预设**
4. **EyeDirection 无目标时的 rotation fallback 优化**

不纳入本轮的内容：

- 多 Overlay 并行叠加架构改造
- 复杂规则表达式系统
- 大规模改造 `PawnFaceConfig` 数据结构
- 彻底事件驱动化的表情系统

## 6. 验证方案

### 6.1 编译验证
执行 CharacterStudio 工程最小构建：

```powershell
dotnet build "..\CharacterStudio\Source\CharacterStudio\CharacterStudio.csproj" -v minimal -nologo
```

### 6.2 功能验证
重点检查：

1. 不同表情下 Overlay 是否会自动切换
2. 多帧整脸表情是否在每个帧边界正确刷新
3. 预览面板是否能一键切换典型状态
4. 无目标时眼睛方向是否能更自然地回退

---

## 7. 后续可选扩展

本轮完成后，可继续推进下一阶段：

### 阶段二候选
- 将 Emotion Overlay 从单值状态升级为多 Overlay 叠加
- 引入短时 reactive expression（受击、惊吓、社交失败等）
- 增加按 mood / pain / combat 的规则触发器
- 在预览界面中增加“缺失资源高亮”和“状态覆盖率检查”

---

## 8. 结论

本方案的核心思路是：

- **不推翻现有架构**
- **优先把已有资源和已有工作流用起来**
- **优先提升创作体验与运行时表现力**
- **用低风险改动，为后续更强的状态驱动面部系统打基础**

本轮完成后，表情模块会从“基础可用”进一步提升到“更易调试、更会表现、更能自动利用资源”的状态。
